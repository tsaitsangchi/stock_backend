## 📜 一、核心治權定義 (The Great Constitution)

本系統之正確性與合法性建立於以下「五維主權」與一項啟動錨點之上。任何未來的修改若偏離此定義，皆判定為「非法變動」：

0. **[Bootstrap Configuration Authority] (啟動錨點條款)**
   - `.env` 是系統架構的 Bootstrap Anchor，負責提供 `PROJECT_ROOT`、執行環境、DB 連線與外部 API token/key 等啟動必要設定。
   - `.env` **不得**成為 25 維路徑清單的 SSOT；它只錨定 `PROJECT_ROOT`，不得展開或複製 `ALL_PATHS`。
   - 密鑰、token、password 僅能存在於 `.env` 或受控秘密管理機制中，禁止寫入程式碼、報告正文或憲章正文。

1. **[Authority of Historical Truth] (歷史權威條款)**
   - 所有程式 Header 必須保留從 v1.0 至今的全量修訂歷程。
   - 歷史紀錄為判定系統「正確性」的唯一基準。

2. **[Hybrid Observability] (混合觀測條款)**
   - 所有維運行為必須觸發「生命週期紀錄」(Execution Log) 與「專項審計紀錄」(Audit Log)。
   - 執行後必須顯示詳細結果摘要（包含診斷狀態、延遲與治權建議）。
   - **任何分支（成功 / 無新資料 / 失敗）皆必須計數並輸出明細，禁止靜默丟失**（v1.17 補強）。
   - `.env` / `path_setup.py` 屬於 pre-schema bootstrap 階段；若 DB schema 尚未建立，允許以 `BOOTSTRAP-DEFERRED` 取代 DB-backed lifecycle/audit 寫入，但必須輸出完整 console report，且不得阻斷路徑自癒。

3. **[Boundary Integrity] (物理邊界條款)**
   - 嚴格遵循 **25 維全譜路徑治理**（`path_setup.py v4.43` 之 `ALL_PATHS`），確保執行環境零配置自癒。
   - `path_setup.py` 是 Path SSOT；所有 25 維物理路徑必須由 `PROJECT_ROOT` 推導，不得由 `.env` 分散定義。
   - `.env` 的 `PROJECT_ROOT` 必須與 `path_setup.py` 計算出的物理根目錄一致；不一致即判定為 Boundary Drift。

4. **[Supply Chain Sovereignty] (供應鏈主權條款)**
   - 數據契約必須 100% 鏡像對齊外部數據源，確保供應鏈穩定。
   - 全譜供應鏈合規稽核必須交叉比對 API + DB 實況（v1.17 補強），不可僅查 API。

5. **[Schema Compatibility Sovereignty] (數據契約相容條約)**
   - **VARCHAR 兼容性**: 實施「防禦性寬容架構」，字串型態欄位統一跳級定義為 `VARCHAR(255)` 或以上。
   - **物理防禦原則**: 確保資料庫 Schema 物理寬度始終「領先且大於」API 數據。
   - **Absolute Case Sovereignty**: 雙引號封裝 DDL，確保物理層與 API 原始大小寫 1:1 鏡像。
   - **Derived Schema 欄位繼承原則**: 未來任何新 table、derived governance table、feature bridge table 或 audit extension，若欄位語意直接引用既有 raw API table，欄位名稱大小寫與 SQL 型別寬度必須與 `data_schema.py` 之 `DATASET_REGISTRY` 對應欄位一致，或採用更寬鬆但不得更窄的相容型別。
   - **治理專用欄位例外**: 只有系統內部治理語意欄位，例如 `snapshot_id`、`policy_version`、`train_eligible`、`selection_reason`，可採 snake_case 命名；不得把 raw API 欄位重新命名為另一種大小寫或縮寫。

6. **[Core Stock Universe Governance] (核心個股宇宙治理條款)**
   - 核心個股不得以主觀名單、硬編程式碼或一次性人工挑選方式定義；必須由 DB-driven、可評分、可重算、可回測、可版本控管之治理流程產生。
   - `TaiwanStockInfo` 為全市場資產清單之基石；核心個股 Universe 必須從此市場宇宙推導，不得繞過市場清單另建孤島。
   - 核心股選拔必須同時反映 `系統核心思想.md` 所定義之第一性原則、20/60/20 厚尾分割、MBNRIC 第六波主題、資料管線可觀察性與數位孿生完整性。
   - 核心股 Universe 是後續特徵、模型、預測與同步的治理輸入，不等同投資建議或買賣清單。

   - 標頭註解必須強制寫入「完整使用範例」，涵蓋以下五大情境：

     - **單一個股所有 Table 同步**

     - **所有核心股 + 所有 Table 強制更新**

### 5.6 混合模式日誌與執行輸出規範 (Hybrid Logging Protocols)

在執行複雜程式、資料同步或腳本任務時，結果呈現方式對於開發者的掌控權至關重要：

1. **混合模式日誌記錄 (Mixed-Mode Logging)**
   - 系統強制要求實作「混合模式」日誌架構：
     - **`pipeline_execution_log`**：記錄完整生命週期（啟動時間、參數、結束狀態）。
     - **專項審計紀錄 (`data_audit_log`)**：記錄局部細節（如寫入筆數、API 重試、錯誤堆疊）。
   - 確保全域監控與局部除錯的需求同時被滿足。

2. **詳細結果展示 (Detailed Result Output)**
   - 程式執行完畢後，必須在標準輸出中顯示「完整且詳細的結果訊息」。

   - 任何資料同步、API 呼叫、DB 寫入若返回空結果或失敗，**禁止靜默結束**。

   - 主權判定必須依實際計數動態計算，**禁止硬編碼 PERFECT**。

## 🏛️ 六、核心個股宇宙治理 (Core Stock Universe Governance)

本章承接 `系統核心思想.md` 之核心哲學思想四：「資料庫驅動的核心標的治理」。自 v5.4.19 起，核心個股不再被視為靜態名單，而是系統從全市場資料、供應鏈契約、交易流動性、基本面重力與第六波主題共振中推導出的治理資產。

### 6.0 核心股治理三層分工 (Authority Layering)

| 程式 | 英文權責 | 中文定位 | 明確邊界 |
| :--- | :--- | :--- | :--- |
| `data_schema.py` | **Raw API Schema Authority** | 原始 API Schema 主權；負責 FinMind/FRED 原始資料表、欄位大小寫、DDL 重鑄與防禦性 schema | 只管理外部 API 原始契約與基礎 log tables；不得建立核心股評分、分層、membership、主題映射等 derived governance tables |
| `core_universe_schema.py` | **Derived Governance Schema Authority** | 衍生治理 Schema 主權；負責開立與維護核心股治理資料層 | 管理 `core_universe_membership`、`core_universe_scores`、`stock_theme_map`、`theme_taxonomy`、`universe_revision_log`，以及 feature/model/prediction eligibility 銜接欄位；不得取代 `data_schema.py` 的 API contract probe；凡引用 raw API 欄位語意者，欄位名稱大小寫與 SQL 型別寬度必須繼承 `DATASET_REGISTRY`；不得保存實際特徵值、label、模型輸出或交易訊號 |
| `core_universe_builder.py` | **Core Universe Selection Authority** | 核心股選拔主權；負責依資料與權重產生核心股 Universe | 只讀取 raw API tables 與治理 tables 後產生評分、分層與 snapshot；不得硬編股票名單，不得自行開立 raw API schema |
| `audit_core_universe.py` | **Core Universe Post-Build Verification** | 核心股結果驗收主權；負責驗收 builder 產物一致性與治理邊界 | 只驗收已產生的 policy/snapshot/membership/scores/revision log、raw 欄位鏡像、downstream boundary 與觀測紀錄；不得重算核心股名單，不得保存特徵值或預測值 |

中文說明：`data_schema.py` 是「原始資料的鏡子」，只負責把 FinMind/FRED 的 table name 與 field name 依 API 原樣落地；`core_universe_schema.py` 是「治理容器」，負責保存核心股選拔所需的 membership、分數、主題、版本、生效期間，以及銜接特徵、模型、預測流程所需的 eligibility / cutoff / coverage / version 欄位；`core_universe_builder.py` 是「選拔引擎」，負責依照資料品質、流動性、基本面、主題共振與風險懲罰重算 Universe；`audit_core_universe.py` 是「驗收稽核」，負責確認 builder 產物已完成 snapshot/membership/scores 一致性與 downstream boundary 封印。`sovereign_sync_engine.py --universe core` 僅能消費治理層已產生的 membership，不得自行決定核心股名單。
### 6.0A Derived Schema 欄位繼承原則 (Column Inheritance Rule)

凡 `core_universe_schema.py` 或未來任何新建 table 需要保存既有 raw API table 的欄位語意，必須遵守以下規則：

1. **名稱繼承**：若欄位來自 `TaiwanStockInfo`、`TaiwanStockPrice`、`TaiwanStockMonthRevenue`、`FredData` 等 raw API table，必須沿用 `data_schema.py` / `DATASET_REGISTRY` 中的原始欄位名稱與大小寫，例如 `stock_id`、`stock_name`、`industry_category`、`Trading_money`、`Trading_Volume`、`revenue`、`series_id`。
2. **型別繼承**：若欄位保存 raw API 同一語意值，SQL 型別必須與 `DATASET_REGISTRY` 對應欄位一致；如需防禦性擴張，只能更寬，不能更窄。例如 raw 為 `VARCHAR(255)`，derived table 不得改成 `VARCHAR(20)`；raw 為 `NUMERIC(20, 6)`，derived table 不得改成 `INTEGER`。
3. **治理欄位分離**：`snapshot_id`、`policy_version`、`core_tier`、`core_score`、`train_eligible`、`predict_eligible`、`selection_reason` 等屬於系統治理語意，可使用 snake_case，因其不是外部 API 欄位鏡像。
4. **禁止語意改名**：不得把 raw 欄位 `Trading_money` 在新表中改名為 `trading_money`，不得把 `PER` 改為 `per`，也不得把 `stock_id` 改為 `symbol`。若需要衍生欄位，必須另命名並清楚標示其為 derived metric。
5. **稽核義務**：未來所有 schema 初始化或 audit 程式，都必須能檢查 derived table 中引用 raw 欄位的名稱與型別是否與 `DATASET_REGISTRY` 相容。

### 6.1 法理來源與治理目的

1. **第一性原則**：核心股必須能被市場物理量解釋，包含流動性質量、資訊衝擊、價格位移、波動承載與資料完整性。
2. **20/60/20 厚尾分割**：系統必須將全市場分成左側風險尾部、中間觀察層與右側正向凸性層，避免只憑主觀偏好挑選標的。
3. **MBNRIC 與第六波主題**：核心 Universe 必須保留醫療、生技、奈米、機器人、資訊/AI、認知科學，以及能源儲存、算力基礎設施、供應鏈樞紐等主題標籤。
4. **DB-driven 標的治理**：核心股成員資格、分數、分層、理由、版本與生效期間必須存在資料庫或受控治理檔，不得散落於程式碼常數。

### 6.5 反偏誤與可回測要求

1. **禁止未來函數**：模型訓練與回測必須使用當時版本的 Universe snapshot，不得用未來選出的核心股回填過去。
2. **禁止倖存者偏誤**：`research_universe` 必須保留足夠寬度，不能只保留後來表現良好的標的。
3. **禁止同步引擎硬編名單**：`sovereign_sync_engine.py --universe core` 必須讀取治理層產生的 membership，而不是在程式碼內寫死股票代號。
4. **必須可觀測**：每一次 Universe 重算、升降級、剔除與啟用，皆需接入 lifecycle / audit log，輸出 PASS / WARN / FAIL 與版本摘要。
5. **必須可重算**：同一份輸入資料、同一組權重版本、同一個基準日期，必須可產生一致的核心股 Universe 結果。

### 6.6 特徵 / 模型 / 預測銜接邊界 (Downstream Bridge Boundary)

核心股治理資料層必須預先考量下游 feature、model、prediction、backtest 的使用需求，但只能保存「治理銜接欄位」，不得保存實際特徵值、訓練 label、模型輸出或交易訊號。

| 類別 | 可存於 `core_universe_schema.py` 治理表 | 不可存於核心股治理表 |
| :--- | :--- | :--- |
| 特徵銜接 | `feature_set_version`、`source_data_cutoff`、`price_coverage_252d`、`revenue_coverage_24m`、`financial_coverage_8q`、`min_history_days` | `MA5`、`RSI`、`MACD`、因子值、標準化後特徵矩陣 |
| 模型銜接 | `model_policy_version`、`train_eligible`、`backtest_eligible`、`label_horizon` | 實際 label、`future_return_20d`、模型權重、訓練參數、模型 artifact |
| 預測銜接 | `prediction_policy_version`、`predict_eligible`、`downstream_ready` | `prediction_score`、預測報酬、買賣訊號、部位建議 |
| 回測銜接 | `snapshot_id`、`as_of_date`、`effective_from`、`effective_to`、Universe membership | 用未來資料回填的 Universe、未來才知道的勝負標籤 |

治理原則如下：

1. **核心股治理層只回答資格問題**：某檔股票在某個 `as_of_date`、某個 `policy_version` 下，是否具備進入特徵生成、模型訓練、預測或回測的資格。
2. **Feature Store 保存實際特徵值**：技術指標、量價因子、基本面因子、籌碼因子與標準化後特徵矩陣，應存於 `data/features`、`data/feature_store` 或未來專屬 feature tables。
3. **Model Registry 保存模型資訊**：模型版本、訓練參數、權重、scaler、artifact 與訓練結果，應歸屬 `models` / `models/mlflow` 或未來模型治理表。
4. **Prediction Layer 保存預測結果**：預測分數、預測報酬、分類結果、交易訊號與推論批次，應歸屬 `inference/predictions` 或未來 prediction tables。
5. **禁止混層污染**：`core_universe_schema.py` 可以為下游流程預留 `eligibility` 與 `version` 欄位，但不得變成 feature store、label store、model registry 或 prediction table。
6. **欄位繼承一致性**：核心股治理表若保存 `stock_id`、`stock_name`、`industry_category` 等 raw API 欄位語意，必須沿用 `data_schema.py` 建立時的欄位大小寫與 SQL 型別相容規則。
