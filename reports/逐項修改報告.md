## 二、核心模組（`scripts/core/`）

### 1. `core/finmind_client.py` — 速率限制與非同步重構

**新增 — `TokenBucketRateLimiter` 類別**

- 依 FinMind 官方配額（每小時 600 次）動態管理令牌。
- 桶容量 = `rate_limit`（預設 600），補充速率 = `rate_limit / period`（每秒 0.167）。
- `min_interval = 0.5s`：兩次請求最小間隔，防止瞬間爆發。
- `acquire()`：執行緒安全的同步令牌消耗。
- `acquire_async()`：協程友善的非阻塞令牌消耗。
- 桶空時帶**抖動因子**（`random.uniform(0, 0.3)`）的退避，
  防止多協程在同一毫秒同時甦醒造成連線風暴；退避基數 1s → 30s 上限。
- 全域單例 `_rate_limiter`，由 `get_rate_limiter()` 對外提供。

**改寫 — `finmind_get()`**

- 新增 `use_rate_limiter=True` 參數（預設啟用）。
- 啟用時：`time.sleep(delay)` 以 `_rate_limiter.acquire()` 取代，
  配額充裕時全速請求、配額緊張時自動降速。
- 退避路徑加入 `random.uniform(0, 1.0)` 抖動因子。
- `delay` 參數保留向後相容（`use_rate_limiter=False` 時仍使用固定延遲）。

**新增 — `finmind_get_async()`**

- 基於 aiohttp 的協程版本，配合 `asyncio.gather` 可並行調度數百個請求。
- 與全域 `TokenBucketRateLimiter` 整合（`acquire_async()`）。
- 402 配額耗盡改用 `await asyncio.sleep`，不阻塞事件迴圈。
- 介面設計：呼叫端傳入 `aiohttp.ClientSession` 以複用 TCP 連線（HTTP Keep-Alive）。

**改寫 — `wait_until_next_hour()`**

- 等待後自動將令牌桶重置為滿載，呼應 FinMind 整點配額重置語意。

### 2. `core/db_utils.py` — 非同步資料庫驅動

**新增 — `get_asyncpg_pool(min_size=2, max_size=10)`**

- 全域 asyncpg 連線池單例（首次呼叫時建立，後續重用）。
- asyncpg 直接實作 PostgreSQL 二進位協定，相較 psycopg2 字串型協定提升 3–5 倍。
- 含 `asyncio.Lock` 確保並行初始化安全。
- 新增 `close_asyncpg_pool()` 供程式結束時清理。

**新增 — `async_bulk_upsert(pool, sql, rows)`**

- 適用中小型資料集的協程版批次寫入。
- 使用 `$1, $2, ...` 佔位符與 `executemany`。

**新增 — `async_bulk_copy_upsert(pool, staging_table, target_table, columns, rows, conflict_columns, update_columns)`**

- 對應優化報告之「二進位暫存表 + 單一合併語句」模式。
- 流程：
  1. `CREATE TEMP TABLE ... LIKE target ON COMMIT DROP`（無約束、SESSION 結束自動清理）
  2. `copy_records_to_table()` 以二進位串流高速複製
  3. 單一 `INSERT ... SELECT ... ON CONFLICT DO UPDATE` 合併至目標表
- 在交易包裹內執行，保證一致性。
- 大型衍生品明細寫入耗時可降低約 70%。

**保留 — 所有同步 API（向後相容）**

- `get_db_conn()`、`ensure_ddl()`、`bulk_upsert()`、`safe_*()`、`get_all_safe_starts()`、
  `resolve_start_cached()`、`dedup_rows()`、`get_db_stock_ids()` 等皆維持原介面。

## 四、未實作項目與後續規劃

優化報告第二部之以下建議**未在本次 commit 落實**，原因與後續路徑說明如下：

| 報告建議 | 未實作原因 | 後續方案 |
|----------|------------|----------|
| **Pydantic 綱要驗證 + 死信佇列** | 需新增 `pydantic` 依賴並重新規劃所有資料模型；與本次「不破壞業務邏輯」原則衝突 | 建議獨立 PR，於 `core/schemas/` 建立 Pydantic 模型，於 fetcher 寫入前統一驗證 |
| **將既有 fetchers 全面改為 asyncio** | 22 支腳本同步重構風險過高；須逐步驗證並監控正確性 | 已具備基礎設施（`finmind_get_async`、`async_bulk_copy_upsert`），可先選定 1–2 支高 I/O 量腳本（如 `fetch_technical_data.py`、`fetch_extended_derivative_data.py`）做 PoC |
| **FinMind SDK 原生 `use_async=True`** | 目前直接以 `requests` 對 REST 端點呼叫；切換至官方 SDK 需重寫請求層 | 待 async PoC 完成後評估，若 SDK 行為與既有重試/402 等待語意一致，可平移 |
| **第三部之新資料源**（TDCC、ETF、PTT、FRED 進階指標） | 屬「資料維度擴張」非本次「底層優化」範疇 | 各自獨立 PR：`fetch_tdcc_distribution.py`、`fetch_etf_holdings.py`、`fetch_ptt_sentiment.py` |

## 五、相容性與測試建議

- **完全向後相容**：所有 fetcher 的 CLI 參數、環境變數需求、寫入的資料表結構、
  日誌輸出格式皆未變動。既有的 `parallel_fetch.py`、`automate_daily.py`、`backfill_from_gaps.py`
  以及下游特徵工程模組無須調整。
- **效能監測點**：
  - Token Bucket 啟用後，建議觀察 `parallel_fetch.py` 完整抓取一輪的總耗時，
    對比靜態 `time.sleep` 的舊版（預期同等或略快，且 402 觸發頻率下降）。
  - asyncpg 路徑為新增 API，不影響既有同步寫入行為；如要試用，
    可在新腳本中以 `await async_bulk_copy_upsert(...)` 取代 `bulk_upsert(...)`。
- **依賴新增**：`aiohttp`、`asyncpg` 為非同步路徑所需，**僅在呼叫對應 async 函式時匯入**
  （延遲匯入），現有同步呼叫鏈路無此依賴需求。如將進一步落實 async 重構，
  應於 `requirements.txt` 補上：
  ```
  aiohttp>=3.9
  asyncpg>=0.29
  ```
