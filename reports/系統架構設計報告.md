# 現代化量化股價預測系統核心架構設計總覽
> **基於 Ubuntu 24.04 與 PostgreSQL 17 的完美實踐**

---

## 第一章：系統願景與架構典範轉移

在構建針對個股未來股價的預測系統時，架構設計的考量點與高頻即時交易系統有著本質上的差異。本系統的核心使命不再是追求微秒級的網路延遲或處理逐筆撮合（Tick）的訂單簿數據，而是轉向如何建立一個具備高度可擴展性、數據一致性與強大運算能力的 **「批量預測與機器學習管線（Machine Learning Pipeline）」**。

現代的股價預測已不再僅僅依賴傳統的統計模型（如 ARIMA），而是全面擁抱機器學習與深度學習架構。透過整合 **長短期記憶網路（LSTM）**、**Transformer** 以及處理非結構化數據的 **大語言模型（LLM）**，系統能夠捕捉股票市場中高度非線性與動態變化的特徵。

為了支撐這些複雜的 AI 運算，一個完美的預測系統架構必須建立在穩固的基礎設施之上。本報告將以 **Ubuntu 24.04** 作為底層作業系統，**PostgreSQL 17** 作為核心數據庫，並以 **Python** 及其強大的數據科學生態系作為運算引擎。

## 第二章：領域驅動設計（DDD）與預測系統的邊界劃分

在股價預測系統中，領域驅動設計（DDD）的重點在於精確劃分資料流與模型生命週期的邊界，確保系統各元件的職責單一且可獨立演進。系統應被嚴格劃分為以下四個核心邊界上下文：

1.  **數據攝取與標準化層 (Data Ingestion Context)**：
    *   負責對接外部數據源（如 FinMind API）。
    *   處理每日收盤後的批量數據（Batch data），包含日 K 線（OHLCV）、基本面財報與籌碼面數據。
2.  **特徵工程與儲存層 (Feature Engineering Context)**：
    *   將原始數據轉化為機器學習模型可用的特徵向量（如技術指標、情緒分數）。
    *   維護特徵一致性，避免訓練與推理時發生 **特徵偏移 (Training-serving skew)**。
3.  **模型訓練與推理層 (Model Training & Inference Context)**：
    *   封裝深度學習演算法，負責定期使用歷史數據進行模型重訓。
    *   在每日數據更新後生成對未來的股價預測信號。
4.  **預測評估與可觀測層 (Evaluation & Observability Context)**：
    *   負責將模型「過去的預測值」與「真實發生的股價」進行對齊比對。
    *   計算預測準確率、誤差，並監控模型漂移。

## 第三章：作業系統與排程基礎設施：Ubuntu 24.04 深度調校

### 3.1 記憶體與資料庫效能調校
在處理數百萬筆歷史 K 線與訓練深度學習模型時，作業系統的記憶體管理至關重要。為了避免 PostgreSQL 在處理龐大查詢時因內核合併記憶體分頁而產生不可預測的延遲突增，系統架構必須在 Ubuntu 層級 **強制關閉透明大頁（Transparent Huge Pages, THP）**。

### 3.2 採用 Systemd Timers 取代 Cron
完美架構將全面採用 Ubuntu 24.04 原生的 **Systemd Timers**。這帶來了三大優勢：
*   **精確的相依性管理**：可設定模型預測服務必須在數據攝取服務成功完成後才啟動。
*   **原生日誌整合**：所有排程任務的標準輸出與錯誤會自動整合進 `journalctl`。
*   **遺漏任務追趕 (Missed Jobs Handling)**：透過設定 `Persistent=true`，確保在意外斷電後自動補跑任務。

## 第四章：後端資料庫架構：PostgreSQL 17 的量化特化實踐

### 4.1 BRIN 索引與時間序列存儲最佳化
針對具備強烈時間順序（Append-only based on time）的股價數據，全面採用 **BRIN（Block Range Indexes）索引**。
*   **優勢**：查詢效能媲美 B-Tree，但能節省高達 **99% 的磁碟空間**，大幅減輕 I/O 負擔。

### 4.2 JSON_TABLE 函數解析複雜數據
PostgreSQL 17 引入了符合 SQL 標準的 **JSON_TABLE** 函數。
*   **優勢**：將外部 API 返回的嵌套 JSON 直接映射為關聯式虛擬資料表，大幅簡化特徵轉換管線。

### 4.3 pgvector 與 LLM 情緒特徵整合
透過 **pgvector** 擴充模組，將大語言模型生成的「文本情緒向量（Embeddings）」直接存入資料庫，達成股價數據與 AI 向量數據的統一存儲。

## 第五章：特徵工程與 Python 核心的高效能架構

### 5.1 採用 Polars 進行極速特徵工程
完美架構將以 **Polars** 取代 Pandas。
*   **優勢**：底層以 Rust 撰寫，支援多執行緒平行運算與 SIMD 指令集，運算速度提升數十倍。

### 5.2 Python 3.12+ 子直譯器（Subinterpreters）
利用 Python 3.12 引入的 **Per-interpreter GIL** 機制，在單一進程內真正平行執行多個股票池的特徵計算，達成極高併發吞吐量。

## 第六章：批量數據攝取策略與 API 整合

系統採用 **FinMind API** 作為外部數據源範例。攝取模組必須實作以下機制：
*   **智能分批與背壓機制 (Backpressure)**：監控 API 使用率，避免觸發速率限制。
*   **增量更新 (Incremental Update)**：僅請求最後一筆日期之後的新數據，最大化使用效率。

---

## 第七章：AI 預測引擎與模型架構

建議採用 **混合型深度學習架構**：
*   **CLT 混合模型**：結合 **1D-CNN**（局部模式）、**Transformer**（全局關聯）與 **LSTM**（長時序依賴）。
*   **多目標輸出**：同時預測「上漲機率」與「預期波動率區間」，提供更豐富的決策依據。

## 第八章：預測對齊、漂移監控與 MLOps 矩陣

### 8.1 預測準確度追蹤 (Actual vs. Predicted)
設計專門的評估資料表，自動執行 SQL 關聯比對預測值與真實值，計算 **MAE** 與方向準確率。

### 8.2 模型漂移 (Model Drift) 預警
當監控模組發現預測誤差連續超過標準差，或 **群體穩定性指標 (PSI)** 發生顯著偏移時，自動觸發重新訓練管線。
