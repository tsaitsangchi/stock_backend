## 第七部：Trinity Core v8.3 高可靠性架構與全量稽核體系 (2026-05 最新升級)

在 v8.3 版本中，我們完成了從「資料數量擴張」到「資料品質極致化」的策略轉型。

### 1. 標的宇宙精煉與淨化 (Core Asset Purification)
*   **痛點**：過往系統嘗試抓取全市場 3,000+ 標的，導致大量創新板 (TIB)、興櫃及低流動性股票產生的「數據斷層 (NaN)」污染特徵空間。
*   **優化方案**：
    *   **128 檔核心宇宙**：精確篩選出 128 檔具備高資料完整度、高市值與代表性的標的。
    *   **物理清理機制**：實作 `migrate_stocks_config.py`，自動同步 `config.py` 標的至資料庫，並物理刪除（Hard Purge）非核心標的。
    *   **元數據同步**：在 `stocks` 表中實作 `fetch_basic`, `fetch_chip`, `fetch_fundamental` 等細分旗標，精準控制每一檔標的的抓取行為。

### 2. 全量數據完整性稽核引擎 (Data Integrity Audit Engine)
*   **核心技術**：實作 `check_data_integrity.py`，建立自動化數據品質掃描儀。
*   **稽核維度**：
    *   **Price (價量)**: 檢測最新交易日是否缺漏。
    *   **Chip (籌碼)**: 驗證三大法人數據連續性。
    *   **Fundamental (財報)**: 檢查季度財務報表更新狀態。
    *   **Revenue (營收)**: 確保月營收報告及時性。
*   **自動修復邏輯**：稽核引擎會自動識別「Missing」狀態，並生成回補任務交由 `data_pipeline` 執行。

### 3. 異質數據源架構對齊 (FinMind & FRED Schema Alignment)
*   **標準化對齊**：
    *   **FinMind**：標準化為 `(date, stock_id)` 複合主鍵，支援 `bulk_upsert`。
    *   **FRED**：將總經指標對齊為 `(date, series_id)` 結構，確保與個股數據具備相同的時間軸處理能力。
*   **一致性校準**：建立 `check_schema_consistency.py`，確保生產環境與開發環境的 Table Schema、Index 與 Constraint 完全一致，解決了「開發能跑、上線報錯」的環境差異問題。

### 4. 混合日誌架構 2.0 (Hybrid Logging 2.0)
*   **生命週期日誌 (`pipeline_execution_log`)**：
    *   紀錄「過程」：Task Name, Category (ingestion/pipeline/maintenance), Duration, Rows Processed.
    *   用於效能監控與自動化排程進度追蹤。
*   **狀態快照日誌 (`data_audit_log`)**：
    *   紀錄「狀態」：`health_score`, `missing_days`, `last_checked_at`。
    *   用於視覺化儀表板，提供即時的數據品質熱點圖。

### 5. 終極路徑自癒技術 (Universal Path-Heal Bootstrap)
*   **問題**：在複雜的目錄結構（`scripts/core/`, `scripts/pipeline/`）下，模組互相匯入易觸發 `ModuleNotFoundError`。
*   **解決方案**：在所有核心腳本頂部導入「Bootstrap 區塊」，自動向上搜索 `scripts` 目錄並動態注入 `sys.path`。這使得腳本無論在根目錄還是子目錄執行，皆能精準定位核心配置與工具函式庫。

python scripts/pipeline/data_pipeline.py

```

```
