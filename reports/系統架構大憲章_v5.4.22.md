> **附註：程式執行順序 (Execution Order Note)**
> 本系統合法啟動與維運順序為：
> 0. `.env` 啟動錨點確認：設定 `PROJECT_ROOT=/home/hugo/project/stock_backend`、DB 連線與外部 API token/key。
> 1. 路徑與目錄自癒：執行 `python scripts/core/path_setup.py`，由 Path SSOT 推導 25 維路徑。
> 2. Raw API Schema 主權重鑄：執行 `python scripts/core/data_schema.py --init --force`；離線復原時可加 `--skip-api-contract`。
> 3. 核心股治理 Schema 初始化：執行 `python scripts/core/core_universe_schema.py --init`，先建立 §6.7 所需 governance tables；此步只建 schema，不要求 membership 已有 committed rows。
> 4. DB 前置依賴檢查：執行 `python scripts/core/db_utils.py`，確認 DB 連線、lifecycle/audit log 寫入、public API 匯出與核心股查詢封裝狀態。初始化期間若 §6.7 查詢回 0 筆，允許判定為 WARNING；若缺表、連線失敗或 log 寫入失敗，則為 FAILED，不得進入下一步。
> 5. Schema 後驗收稽核：執行 `python scripts/maintenance/audit_supply_chain.py --include-logs`；離線 DB 驗收可用 `--db-only --include-logs`。此工具會檢查最近 24 小時 pipeline log，任何 failed status 皆會使驗收 FAILED。
> 6. 市場個股資料種子灌溉：執行 `python scripts/ingestion/sovereign_sync_engine.py --seed`。
> 7. 核心股 Universe metadata bootstrap：先執行 `python scripts/core/core_universe_builder.py --dry-run --as-of-date 2026-05-14`，確認後改用 `--commit`。
> 8. 核心股結果驗收稽核：執行 `python scripts/maintenance/audit_core_universe.py --as-of-date 2026-05-14`。
> 9. 同步灌溉操作：依需求執行 `sovereign_sync_engine.py --id 2330`、`--id 2330 --dataset TaiwanStockPrice`、`--universe core` 或 `--source fred`；其中 `--universe core` 必須透過 §6.7 SQL 契約取得核心股名單。

本系統之正確性與合法性建立於以下「五維主權」與一項啟動錨點之上。任何未來的修改若偏離此定義，皆判定為「非法變動」：

0. **[Bootstrap Configuration Authority] (啟動錨點條款)**
   - `.env` 是系統架構的 Bootstrap Anchor，負責提供 `PROJECT_ROOT`、執行環境、DB 連線與外部 API token/key 等啟動必要設定。
   - `.env` **不得**成為 25 維路徑清單的 SSOT；它只錨定 `PROJECT_ROOT`，不得展開或複製 `ALL_PATHS`。
   - 密鑰、token、password 僅能存在於 `.env` 或受控秘密管理機制中，禁止寫入程式碼、報告正文或憲章正文。

1. **[Authority of Historical Truth] (歷史權威條款)**
   - 所有程式 Header 必須保留從 v1.0 至今的全量修訂歷程。
   - 歷史紀錄為判定系統「正確性」的唯一基準。

2. **[Hybrid Observability] (混合觀測條款)**
   - 所有維運行為必須觸發「生命週期紀錄」(Execution Log) 與「專項審計紀錄」(Audit Log)。
   - 執行後必須顯示詳細結果摘要（包含診斷狀態、延遲與治權建議）。
   - **任何分支（成功 / 無新資料 / 失敗）皆必須計數並輸出明細，禁止靜默丟失**（v1.17 補強）。
   - `.env` / `path_setup.py` 屬於 pre-schema bootstrap 階段；若 DB schema 尚未建立，允許以 `BOOTSTRAP-DEFERRED` 取代 DB-backed lifecycle/audit 寫入，但必須輸出完整 console report，且不得阻斷路徑自癒。
   - **(v5.4.20 補強)** `db_utils.record_lifecycle()` 寫入 `pipeline_execution_log` 失敗時（例如表尚未建立），必須降級為 console-only warning，不得阻斷 caller。

3. **[Boundary Integrity] (物理邊界條款)**
   - 嚴格遵循 **25 維全譜路徑治理**（`path_setup.py v4.44` 之 `ALL_PATHS`），確保執行環境零配置自癒。
   - `path_setup.py` 是 Path SSOT；所有 25 維物理路徑必須由 `PROJECT_ROOT` 推導，不得由 `.env` 分散定義。
   - `.env` 的 `PROJECT_ROOT` 必須與 `path_setup.py` 計算出的物理根目錄一致；不一致即判定為 Boundary Drift。

4. **[Supply Chain Sovereignty] (供應鏈主權條款)**
   - 數據契約必須 100% 鏡像對齊外部數據源，確保供應鏈穩定。
   - 全譜供應鏈合規稽核必須交叉比對 API + DB 實況（v1.17 補強），不可僅查 API。

5. **[Schema Compatibility Sovereignty] (數據契約相容條約)**
   - **VARCHAR 兼容性**: 實施「防禦性寬容架構」，字串型態欄位統一跳級定義為 `VARCHAR(255)` 或以上。
   - **物理防禦原則**: 確保資料庫 Schema 物理寬度始終「領先且大於」API 數據。
   - **Absolute Case Sovereignty**: 雙引號封裝 DDL，確保物理層與 API 原始大小寫 1:1 鏡像。
   - **Derived Schema 欄位繼承原則**: 未來任何新 table、derived governance table、feature bridge table 或 audit extension，若欄位語意直接引用既有 raw API table，欄位名稱大小寫與 SQL 型別寬度必須與 `data_schema.py` 之 `DATASET_REGISTRY` 對應欄位一致，或採用更寬鬆但不得更窄的相容型別。
   - **治理專用欄位例外**: 只有系統內部治理語意欄位，例如 `snapshot_id`、`policy_version`、`train_eligible`、`selection_reason`，可採 snake_case 命名；不得把 raw API 欄位重新命名為另一種大小寫或縮寫。

6. **[Core Stock Universe Governance] (核心個股宇宙治理條款)**
   - 核心個股不得以主觀名單、硬編程式碼或一次性人工挑選方式定義；必須由 DB-driven、可評分、可重算、可回測、可版本控管之治理流程產生。
   - `TaiwanStockInfo` 為全市場資產清單之基石；核心個股 Universe 必須從此市場宇宙推導，不得繞過市場清單另建孤島。
   - 核心股選拔必須同時反映 `系統核心思想.md` 所定義之第一性原則、20/60/20 厚尾分割、MBNRIC 第六波主題、資料管線可觀察性與數位孿生完整性。
   - 核心股 Universe 是後續特徵、模型、預測與同步的治理輸入，不等同投資建議或買賣清單。
   - **(v5.4.20 補強)** 任何模組欲取得核心股名單，**必須**透過 §6.7 定義之單一 SQL 契約查 `core_universe_membership`，禁止直接查 v5.2 時代之 `stocks` 表或其他未授權來源。

### 模組憲章對齊註記 (v5.4.21 完整文本版重寫)

## 🏛️ 五、核心定義實體實踐 (Constitution in Practice)

### 5.5 程式碼標頭與邊界保護

1. **核心目錄保護原則 (No-touch Zones)**
   - 絕對遵循「以 `core/` 目錄程式不修改為原則」。核心業務邏輯與架構屬於系統紅線區。
   - 動 `scripts/core/` 任何檔案前，必須得到「明示書面授權」，且升版必須保留全量歷程。

2. **全量修訂歷程標頭 (Comprehensive Revision Header)**
   - 所有程式檔案標頭必須包含：
     - **檔案名稱與目的**
     - **初始建立者與日期**
     - **修訂歷程 (Revision History)**：必須「完整保留」所有修改歷程，絕不可抹除。
   - **標頭宣告版本必須等於 revision history 之 ACTIVE 版本**。

3. **執行程式使用範例說明 (Execution Usage Examples)**
   - 標頭註解必須強制寫入「完整使用範例」，涵蓋以下五大情境：
     - **個股同步**
     - **單一 Table 同步**
     - **單一個股所有 Table 同步**
     - **所有核心股同步**
     - **所有核心股 + 所有 Table 強制更新**

### 5.6 混合模式日誌與執行輸出規範 (Hybrid Logging Protocols)

1. **混合模式日誌記錄 (Mixed-Mode Logging)**
   - 系統強制要求實作「混合模式」日誌架構：
     - **`pipeline_execution_log`**：記錄完整生命週期（啟動時間、參數、結束狀態）。
     - **專項審計紀錄 (`data_audit_log`)**：記錄局部細節（如寫入筆數、API 重試、錯誤堆疊）。

2. **詳細結果展示 (Detailed Result Output)**
   - 程式執行完畢後，必須在標準輸出中顯示「完整且詳細的結果訊息」。
   - 必須包含：成功筆數、錯誤項目、總耗時、以及分類日誌的索引關聯。

3. **零靜默丟失原則 (Zero Silent Drop, v1.7 + v1.17 補強)**
   - 任何資料同步、API 呼叫、DB 寫入若返回空結果或失敗，**禁止靜默結束**。
   - 必須走明確的 success / warning / failed 分支之一，並計入 stats 與 details。
   - 主權判定必須依實際計數動態計算，**禁止硬編碼 PERFECT**。

## 🏛️ 六、核心個股宇宙治理 (Core Stock Universe Governance)

本章承接 `系統核心思想.md` 之核心哲學思想四：「資料庫驅動的核心標的治理」。自 v5.4.19 起，核心個股不再被視為靜態名單，而是系統從全市場資料、供應鏈契約、交易流動性、基本面重力與第六波主題共振中推導出的治理資產。

### 6.0 核心股治理三層分工 (Authority Layering)

| 程式 | 英文權責 | 中文定位 | 明確邊界 |
| :--- | :--- | :--- | :--- |
| `data_schema.py` | **Raw API Schema Authority** | 原始 API Schema 主權 | 只管理外部 API 原始契約與基礎 log tables；不得建立核心股 derived governance tables |
| `core_universe_schema.py` | **Derived Governance Schema Authority** | 衍生治理 Schema 主權 | 管理核心股治理表；不得取代 `data_schema.py` 的 API contract probe |
| `core_universe_builder.py` | **Core Universe Selection Authority** | 核心股選拔主權 | 只讀取 raw API tables 與治理 tables 後產生評分、分層與 snapshot；不得硬編股票名單 |
| `audit_core_universe.py` | **Core Universe Post-Build Verification** | 核心股結果驗收主權 | 只驗收已產生的 snapshot；不得重算核心股名單 |

### 6.0A Derived Schema 欄位繼承原則 (Column Inheritance Rule)

凡引用既有 raw API table 欄位語意者，欄位名稱大小寫與 SQL 型別寬度必須繼承 `DATASET_REGISTRY`，治理專用欄位才可另立 snake_case。完整規則沿用 v5.4.19 6.0A 條文，本版不修訂。

### 6.1 ~ 6.6 法理 / 分層 / CoreScore / 治理欄位 / 反偏誤 / 銜接邊界

完整法理沿用 v5.4.19 條文；本版補明「治理藍圖」與「現行實作版本」之邊界，避免將 v0.1 metadata bootstrap 誤認為完整核心股選拔模型。核心原則為：**先建立選拔制度，再產生名單；先選資料條件，不先選股票代號。**

- **6.1 法理**：第一性原則、20/60/20、MBNRIC、DB-driven。
- **6.2 分層**：research_universe / core_universe / convex_universe / quarantine_universe。
- **6.3 CoreScore**：25% DataQuality + 25% LiquidityMass + 20% FundamentalGravity + 15% ThemeResonance + 10% InstitutionalFlow + 5% VolatilityControl − RiskPenalty。
- **6.4 治理欄位**：含 price_coverage_252d、revenue_coverage_24m、financial_coverage_8q 等 lookback 指標。
- **6.5 反偏誤**：禁未來函數、禁倖存者偏誤、**禁同步引擎硬編名單**、必須可觀測、必須可重算。
- **6.6 銜接邊界**：治理層不存 feature/label/model/prediction values。

**核心股建立標準流程**：

1. `core_universe_schema.py`：先建立核心股治理 tables，只負責 governance schema，不碰 `data_schema.py` 的 raw API schema。
2. 歷史資料灌溉：取得價格、成交量、月營收、PER、法人、融資融券、財報等資料，作為 CoreScore v0.2 的輸入。
3. `core_universe_builder.py`：依資料計算 CoreScore、分層、snapshot、membership、scores 與 revision log。
4. `audit_core_universe.py`：驗收 Universe 是否合理、可回測、無偏誤、無跨層污染。
5. `sovereign_sync_engine.py --universe research / convex / core`：研究宇宙分階段灌溉先消費 `research_universe`，再視需要補 `convex_universe`，最後才由 `core` 消費已 committed 的 membership；同步引擎不得自行決定核心股名單。

**CoreScore v0.2 六層評分流程**：

1. **硬性資格過濾**：
   - 必須存在於 `TaiwanStockInfo`。
   - 排除資料缺漏嚴重標的。
   - 排除極低流動性標的。
   - 排除長期停牌、異常交易、風險過高標的。

2. **資料品質分數 (DataQuality)**：
   - 價格資料完整度。
   - 成交量資料完整度。
   - 月營收資料完整度。
   - 財報資料完整度。
   - API / DB 欄位一致性。

3. **流動性質量分數 (LiquidityMass)**：
   - 平均日成交金額。
   - 平均成交量。
   - 成交連續性。
   - 波動承載能力。
   - 價格跳空與極端缺口。

4. **基本面重力分數 (FundamentalGravity)**：
   - 月營收成長。
   - 獲利能力。
   - 現金流或財報穩定度。
   - PER 合理性。
   - 股利與財務韌性。

5. **主題共振分數 (ThemeResonance)**：
   - AI / 半導體 / 算力供應鏈。
   - 機器人 / 自動化。
   - 生技 / 醫療。
   - 能源 / 儲能。
   - MBNRIC 第六波主題。

6. **風險懲罰 (RiskPenalty)**：
   - 高波動但無流動性支撐。
   - 基本面惡化。
   - 資料缺失。
   - 籌碼異常。
   - 長期弱勢或左尾風險。

**CoreScore v0.2 最小輸入資料契約（8 類）**：

`core_universe_builder.py v0.2` 的輸入資料以 `scripts/core/data_schema.py` 的 `DATASET_REGISTRY` 為唯一 schema 來源。表名、欄位名與唯一鍵不得在 builder 內另行定義；若 API 回傳欄位與 DB 欄位不一致，必須先在 ingestion / schema 對齊層修正，不得在選股模型內以臨時欄位補丁處理。

| 類別 | 主資料表 | 最小欄位契約 | 主要用途 | v0.2 裁決 |
|---|---|---|---|---|
| 1. 市場清單 | `TaiwanStockInfo` | `stock_id`, `stock_name`, `industry_category`, `type`, `date` | 候選池、產業分類、股票存續檢查 | 不存在於此表者不得進入候選池 |

| 3. 月營收 | `TaiwanStockMonthRevenue` | `date`, `stock_id`, `country`, `revenue`, `revenue_month`, `revenue_year`, `create_time` | 營收成長、基本面趨勢、資料完整度 | 缺漏進入 DataQuality 與 FundamentalGravity 扣分 |
| 4. 估值 | `TaiwanStockPER` | `date`, `stock_id`, `dividend_yield`, `PER`, `PBR` | PER / PBR 合理性、股利收益支撐 | 缺漏不得靜默補值，需寫入 `score_detail` |
| 5. 法人籌碼 | `TaiwanStockInstitutionalInvestorsBuySell` | `date`, `stock_id`, `name`, `buy`, `sell` | 外資、投信、自營商買賣超與籌碼趨勢 | 覆蓋率不足時籌碼分數中立或扣分，不得硬補 |
| 6. 融資融券 | `TaiwanStockMarginPurchaseShortSale` | `date`, `stock_id`, `MarginPurchaseTodayBalance`, `MarginPurchaseYesterdayBalance`, `ShortSaleTodayBalance`, `ShortSaleYesterdayBalance`, `OffsetLoanAndShort`, `Note` | 槓桿風險、散戶擁擠度、風險懲罰 | 異常槓桿變化進入 RiskPenalty |
| 7. 財報 | `TaiwanStockFinancialStatements` | `date`, `stock_id`, `type`, `value`, `origin_name` | 獲利能力、財務穩定度、現金流或科目品質 | `financial_coverage_8q` 是 FundamentalGravity 可用性依據 |
| 8. FRED 宏觀 | `FredData` | `date`, `series_id`, `value`, `realtime_start`, `realtime_end` | 利率、美元、信用、通膨與景氣環境 | 表級缺漏阻止 v0.2 正式評分；個股不因宏觀資料單獨排除 |

v0.2 的標準觀察窗如下：

- 價格、成交量、PER、法人籌碼、融資融券：以 `as_of_date` 向前至少 252 個交易日為核心覆蓋率計算基準，建議保留 730 日供波動與趨勢特徵使用。
- 月營收：以 `as_of_date` 向前 24 個月計算 `revenue_coverage_24m` 與 YoY / MoM 趨勢。
- 財報：以 `as_of_date` 向前 8 季計算 `financial_coverage_8q`。
- FRED：以 `as_of_date` 可取得的 `realtime_start` / `realtime_end` 版本為準，避免使用未來才修正完成的宏觀資料。

### 6.7 核心股查詢 SQL 契約 (Core Stock Retrieval SQL Contract) — *v5.4.20 新增*

本條為 §6.5 第 3 條「禁止同步引擎硬編名單」之具體實作契約。所有模組（含基礎設施 library 與序列模組）若需取得「目前生效之核心股名單」，**必須**使用以下唯一 SQL 契約，**不得**自行另立 SELECT 來源：

```sql
SELECT m."stock_id"
FROM "core_universe_membership" m
JOIN "core_universe_snapshot" s
  ON s."snapshot_id" = m."snapshot_id"
WHERE s."status" = 'committed'
  AND m."core_tier" IN ('core_universe', 'convex_universe')
  AND s."as_of_date" = (
      SELECT MAX("as_of_date")
      FROM "core_universe_snapshot"
      WHERE "status" = 'committed'
  )
ORDER BY m."stock_id";
```

**契約規格**：

1. **唯一來源**：`core_universe_membership` JOIN `core_universe_snapshot`；**禁止**查 v5.2 時代之 `stocks` 表、`TaiwanStockInfo`（市場宇宙）或任何硬編清單。
2. **生效條件**：必須 JOIN `core_universe_snapshot` 並過濾 `status = 'committed'`；未提交的 `draft` snapshot 不得作為來源。
3. **時點選擇**：必須選 `MAX(as_of_date)` 之 committed snapshot；若需歷史時點重算（Walk-forward），caller 應接受 `as_of_date` 參數並改為等號比對。
4. **層級範圍**：預設回傳 `core_universe` ∪ `convex_universe`；`research_universe` 與 `quarantine_universe` 不在預設同步範圍，但若為研究宇宙分階段灌溉，caller 可透過 `tiers=('research_universe',)` 明示納入 stage 1 補刷。
5. **排序**：必須 `ORDER BY stock_id` 以保證冪等與可重算。
6. **空集處理**：若 SELECT 回傳 0 筆，caller 必須以 `warning` 處理（不得 silent succeed），並通知 lifecycle context。

**封裝義務**：

- 本契約**應由 `db_utils.get_core_stocks_from_db()` 封裝**，供所有 caller 共用，避免每個模組各自寫 SQL（違反 DRY 與 SSOT）。
- `db_utils.py v2.45` 已完成封裝，實作查詢來源為 `core_universe_membership` JOIN `core_universe_snapshot`，並以 `status='committed'` 與 `MAX(as_of_date)` 保證生效版本唯一。
- 未來若需擴充（如歷史時點 `as_of_date` 參數、層級篩選參數），應在 `get_core_stocks_from_db()` 增加 keyword arguments，仍維持單一 SQL 來源。

**例外條款**：

- `audit_core_universe.py` 為驗收稽核工具，**允許**直接讀取 `core_universe_membership` 全部 tier 以驗證 builder 產物完整性；但不得回填到 sync 引擎。
- `core_universe_builder.py` 為 selection authority 本身，產生 membership，當然不適用本契約。

## 🏛️ 七、供應鏈速率主權與重試契約 (Supply Chain Rate Sovereignty & Retry Contract) — *2026-05-15 同日補充*

本章為 2026-05-15 實證發現 `sovereign_sync_engine.py v1.9` 連續三次失敗（pipeline id 14/15/16）之治權回應。實證確認失敗根因為「FinMind API 6000 次/小時上限 + 30 分鐘配額重置週期」與「v1.9 缺乏節流、退避、續跑三層機制」之結構性缺口，**非** schema 漂移、token 失效或資料集權限不足。本章將節流、退避、續跑、配額觀測四項主權銘刻為憲法級契約，並指定 `sovereign_sync_engine.py v1.10` 為實作載體。

### 7.1 帳號實況基線 (Account Reality Baseline)

| 項目 | 值 | 說明 |
| :--- | :--- | :--- |
| 每小時請求上限 | 6,000 次 | FinMind 主權公告 |
| 配額重置週期 | 30 分鐘 | 滑動或固定窗，實證以 30 分鐘對齊最穩健 |
| 觸發 403 的訊號 | `HTTP 403 Forbidden` / `HTTP 429 Too Many Requests` | 視為「速率超限」而非「永久拒絕」 |
| 觸發 402 的訊號 | `HTTP 402 Payment Required` | 主要視為「資料集需付費升級」；但容許 FinMind 漂移為「付費方案小時用量超標」之語義可能，故給予**單次 1800s 探測重試**；詳見 §7.4 |

### 7.2 三層防禦契約 (Three-Layer Defense Contract)

任何呼叫 FinMind API 的同步引擎或 fetcher，**必須**同時實作以下三層；缺一層即視為違憲：

| 層級 | 名稱 | 必要條件 | 違憲判定 |
| :--- | :--- | :--- | :--- |
| L1 | **事前預防 (Proactive Throttle)** | 滑動窗節流，配額上限以 **5,500 / hr** 為主權預設（保留 8% 餘裕），單一 worker 串行呼叫 | 無節流或上限 ≥ 6,000 |
| L2 | **事中應對 (Tiered Backoff Retry)** | 分階段退避序列 **`[30s, 300s, 1800s]`**，第三階段必須對齊 30 分鐘重置週期 | 固定 60s、指數退避、或無上限重試 |
| L3 | **事後續跑 (DB-Driven Resume)** | 以 DB 既有資料作為 checkpoint，已有 `(stock_id, dataset, ≥ start_date)` 紀錄者**不得**再次呼叫 API | 重跑時整批重打、或以本地檔案而非 DB 做 checkpoint |

### 7.3 退避序列裁決 (Backoff Sequence Verdict)

退避序列固定為 `[30s, 300s, 1800s]`，總計上限 **35 分鐘**。本序列為憲法級常數，未經主權變更不得自行調整。

| 第 N 次 retry | 等待時間 | 對應情境 | 不可調整的理由 |
| :--- | :--- | :--- | :--- |
| 1 | 30 秒 | 瞬時網路抖動、上游 5xx | 短於 30s 會在 reset 前撞回 |
| 2 | 5 分鐘 | 短擁塞、API 端短暫過載 | 5 分鐘可吸收大多數短期波動 |
| 3 | 30 分鐘 | 配額視窗未重置 | **必須**對齊 FinMind 30 分鐘重置週期，提早醒會立即再次 403 |
| 4+ | — | 不重試，直接 FAILED 並寫入 lifecycle | 連續三次失敗代表非速率問題，需人工介入 |

### 7.4 HTTP 狀態碼處理矩陣 (HTTP Status Handling Matrix)

| 狀態碼 | 語意 | 處理動作 | 是否計入 retry |
| :--- | :--- | :--- | :--- |
| 200 | 成功 | 正常處理 payload | 否 |
| 402 | Payment Required | **單次 1800s 探測重試**（容許 FinMind 把「付費方案小時用量超標」漂移為 402 的可能性）；第 2 次仍 402 即 FAILED，寫入 `data_audit_log` 標註「資料集付費門檻」；第 2 次成功則以 `402-recovered` 標籤紀錄於 audit log，供後續配額語義統計 | 是（單次） |
| 403 | Forbidden（速率超限） | 進入 L2 完整退避序列 `[30s, 300s, 1800s]` | 是（三次） |
| 429 | Too Many Requests | 進入 L2 完整退避序列 `[30s, 300s, 1800s]` | 是（三次） |
| 5xx | 上游伺服器錯誤 | 進入 L2 完整退避序列 `[30s, 300s, 1800s]` | 是（三次） |
| 4xx 其他 | 客戶端錯誤 | 立即 FAILED，寫入 lifecycle，不重試 | 否 |
| Timeout | 連線/讀取逾時 | 進入 L2 完整退避序列 `[30s, 300s, 1800s]` | 是（三次） |

> **402 與 403 的關鍵差異（修訂版）**：402 在標準 HTTP 語義為「永久付費門檻」，但 FinMind 等廠商可能把「付費方案的小時用量超標」漂移為 402；為兼顧兩種情境，**402 容許單次 1800s 探測重試**——若仍 402 即視為情境 A（永久付費門檻）並立刻放棄；若 200 即視為情境 B（漂移型 402，等同 quota 重置）並繼續同步。403/429 則為明確速率超限，必須走完整三階段退避。**禁止**對 402 使用多階段退避（總成本不應超過 1 次浪費 + 30 分鐘等待）。
>
> **`402-recovered` audit 標籤**：每次 402 重試成功者必須於 `data_audit_log` 寫入 `op_type='RETRY_402_RECOVERED'`，累積至 ≥ 30 筆樣本後可作為 §7.4 條文升版證據，屆時若統計顯示 FinMind 402 ≥ 90% 屬漂移型，可主權升版允許 402 進入完整 L2 退避序列。

### 7.5 斷點續傳資料源契約 (Resume Data Source Contract)

L3 斷點續傳的 checkpoint 來源**唯一**為 DB，禁止使用本地檔案、Redis、cache 或記憶體 dict：

```sql
SELECT 1 FROM "<dataset_table>"
WHERE "stock_id" = %s AND "date" >= %s
LIMIT 1;
```

- **唯一性**：以 `(stock_id, dataset, >= start_date)` 為 idempotent key，存在即視為已同步。
- **可重跑性**：任何中斷後重跑均可立即跳過已完成項，**禁止**重打 API。
- **與 §5.6 零靜默丟失相容**：跳過動作必須走 `warning` 或 `info` 分支寫入 stats/details，禁止 silent skip。

### 8.5 Data Leakage 防禦規則 (Anti-Leakage Defense Rules)

| 規則 | 適用層 | 違例範例 | 強制執行載體 |
|------|--------|---------|-------------|
| **as-of-strict filter** | Feature / Model / Prediction | `WHERE date <= as_of_date + N` (N>0) | builder/trainer/engine 必須通過 `audit_leakage.py` |
| **label horizon 後置** | Model | 用 `t` 日 features 預測 `t+0` return | trainer 必須驗證 `label_date >= as_of_date + label_horizon` |
| **universe snapshot 鎖定** | All | 訓練用 v0.2 snapshot, 推論用 v0.3 snapshot 而未重訓 | 三層 DDL 之 `universe_snapshot_id` 必須一致 |
| **feature_set 鎖定** | Model / Prediction | 訓練用 fs_v0.1, 推論用 fs_v0.2 | `model_registry.feature_set_id` 必須等於 `prediction_run.feature_set_id` |
| **No hot-fix imputation** | Feature | 推論時補 NaN 用「整個歷史平均」 | `feature_definition.null_strategy` 必須在 build 時宣告，推論時禁止改變 |
| **No future split** | Feature | 用後續資料計算的標準化 mean/std | 標準化參數必須鎖在 `feature_definition` 或 train artifact |
| **零硬編預測** | Prediction | 在 .py 中硬編 `if stock_id=='2330': return 0.05` | `audit_supply_chain.py` 延伸掃描 |
| **單一 SSOT**（§6.7 延伸） | All | 直接 SELECT TaiwanStockInfo 取核心股 | 必須走 `db_utils.get_core_stocks_from_db()` |
