## 📜 一、核心治權定義 (The Great Constitution)

本系統之正確性與合法性建立於以下「五維主權」與一項啟動錨點之上。任何未來的修改若偏離此定義，皆判定為「非法變動」：

0. **[Bootstrap Configuration Authority] (啟動錨點條款)**
   - `.env` 是系統架構的 Bootstrap Anchor，負責提供 `PROJECT_ROOT`、執行環境、DB 連線與外部 API token/key 等啟動必要設定。
   - `.env` **不得**成為 25 維路徑清單的 SSOT；它只錨定 `PROJECT_ROOT`，不得展開或複製 `ALL_PATHS`。
   - 密鑰、token、password 僅能存在於 `.env` 或受控秘密管理機制中，禁止寫入程式碼、報告正文或憲章正文。

1. **[Authority of Historical Truth] (歷史權威條款)**
   - 所有程式 Header 必須保留從 v1.0 至今的全量修訂歷程。
   - 歷史紀錄為判定系統「正確性」的唯一基準。

2. **[Hybrid Observability] (混合觀測條款)**
   - 所有維運行為必須觸發「生命週期紀錄」(Execution Log) 與「專項審計紀錄」(Audit Log)。
   - 執行後必須顯示詳細結果摘要（包含診斷狀態、延遲與治權建議）。
   - **任何分支（成功 / 無新資料 / 失敗）皆必須計數並輸出明細，禁止靜默丟失**（v1.17 補強）。
   - `.env` / `path_setup.py` 屬於 pre-schema bootstrap 階段；若 DB schema 尚未建立，允許以 `BOOTSTRAP-DEFERRED` 取代 DB-backed lifecycle/audit 寫入，但必須輸出完整 console report，且不得阻斷路徑自癒。
   - **(v5.4.20 補強)** `db_utils.record_lifecycle()` 寫入 `pipeline_execution_log` 失敗時（例如表尚未建立），必須降級為 console-only warning，不得阻斷 caller。

3. **[Boundary Integrity] (物理邊界條款)**
   - 嚴格遵循 **25 維全譜路徑治理**（`path_setup.py v4.44` 之 `ALL_PATHS`），確保執行環境零配置自癒。
   - `path_setup.py` 是 Path SSOT；所有 25 維物理路徑必須由 `PROJECT_ROOT` 推導，不得由 `.env` 分散定義。
   - `.env` 的 `PROJECT_ROOT` 必須與 `path_setup.py` 計算出的物理根目錄一致；不一致即判定為 Boundary Drift。

4. **[Supply Chain Sovereignty] (供應鏈主權條款)**
   - 數據契約必須 100% 鏡像對齊外部數據源，確保供應鏈穩定。
   - 全譜供應鏈合規稽核必須交叉比對 API + DB 實況（v1.17 補強），不可僅查 API。

5. **[Schema Compatibility Sovereignty] (數據契約相容條約)**
   - **VARCHAR 兼容性**: 實施「防禦性寬容架構」，字串型態欄位統一跳級定義為 `VARCHAR(255)` 或以上。
   - **物理防禦原則**: 確保資料庫 Schema 物理寬度始終「領先且大於」API 數據。
   - **Absolute Case Sovereignty**: 雙引號封裝 DDL，確保物理層與 API 原始大小寫 1:1 鏡像。
   - **Derived Schema 欄位繼承原則**: 未來任何新 table、derived governance table、feature bridge table 或 audit extension，若欄位語意直接引用既有 raw API table，欄位名稱大小寫與 SQL 型別寬度必須與 `data_schema.py` 之 `DATASET_REGISTRY` 對應欄位一致，或採用更寬鬆但不得更窄的相容型別。
   - **治理專用欄位例外**: 只有系統內部治理語意欄位，例如 `snapshot_id`、`policy_version`、`train_eligible`、`selection_reason`，可採 snake_case 命名；不得把 raw API 欄位重新命名為另一種大小寫或縮寫。

6. **[Core Stock Universe Governance] (核心個股宇宙治理條款)**
   - 核心個股不得以主觀名單、硬編程式碼或一次性人工挑選方式定義；必須由 DB-driven、可評分、可重算、可回測、可版本控管之治理流程產生。
   - `TaiwanStockInfo` 為全市場資產清單之基石；核心個股 Universe 必須從此市場宇宙推導，不得繞過市場清單另建孤島。

## 🏛️ 五、核心定義實體實踐 (Constitution in Practice)

### 5.5 程式碼標頭與邊界保護

1. **核心目錄保護原則 (No-touch Zones)**
   - 絕對遵循「以 `core/` 目錄程式不修改為原則」。核心業務邏輯與架構屬於系統紅線區。
   - 動 `scripts/core/` 任何檔案前，必須得到「明示書面授權」，且升版必須保留全量歷程。

2. **全量修訂歷程標頭 (Comprehensive Revision Header)**
   - 所有程式檔案標頭必須包含：
     - **檔案名稱與目的**
     - **初始建立者與日期**
     - **修訂歷程 (Revision History)**：必須「完整保留」所有修改歷程，絕不可抹除。
   - **標頭宣告版本必須等於 revision history 之 ACTIVE 版本**。

3. **執行程式使用範例說明 (Execution Usage Examples)**
   - 標頭註解必須強制寫入「完整使用範例」，涵蓋以下五大情境：
     - **個股同步**
     - **單一 Table 同步**
     - **單一個股所有 Table 同步**
     - **所有核心股同步**
     - **所有核心股 + 所有 Table 強制更新**

### 5.6 混合模式日誌與執行輸出規範 (Hybrid Logging Protocols)

1. **混合模式日誌記錄 (Mixed-Mode Logging)**
   - 系統強制要求實作「混合模式」日誌架構：
     - **`pipeline_execution_log`**：記錄完整生命週期（啟動時間、參數、結束狀態）。
     - **專項審計紀錄 (`data_audit_log`)**：記錄局部細節（如寫入筆數、API 重試、錯誤堆疊）。

2. **詳細結果展示 (Detailed Result Output)**
   - 程式執行完畢後，必須在標準輸出中顯示「完整且詳細的結果訊息」。
   - 必須包含：成功筆數、錯誤項目、總耗時、以及分類日誌的索引關聯。

3. **零靜默丟失原則 (Zero Silent Drop, v1.7 + v1.17 補強)**
   - 任何資料同步、API 呼叫、DB 寫入若返回空結果或失敗，**禁止靜默結束**。
   - 必須走明確的 success / warning / failed 分支之一，並計入 stats 與 details。
   - 主權判定必須依實際計數動態計算，**禁止硬編碼 PERFECT**。

## 🏛️ 六、核心個股宇宙治理 (Core Stock Universe Governance)

本章承接 `系統核心思想.md` 之核心哲學思想四：「資料庫驅動的核心標的治理」。自 v5.4.19 起，核心個股不再被視為靜態名單，而是系統從全市場資料、供應鏈契約、交易流動性、基本面重力與第六波主題共振中推導出的治理資產。

### 6.0 核心股治理三層分工 (Authority Layering)

| 程式 | 英文權責 | 中文定位 | 明確邊界 |
| :--- | :--- | :--- | :--- |
| `data_schema.py` | **Raw API Schema Authority** | 原始 API Schema 主權 | 只管理外部 API 原始契約與基礎 log tables；不得建立核心股 derived governance tables |
| `core_universe_schema.py` | **Derived Governance Schema Authority** | 衍生治理 Schema 主權 | 管理核心股治理表；不得取代 `data_schema.py` 的 API contract probe |
| `core_universe_builder.py` | **Core Universe Selection Authority** | 核心股選拔主權 | 只讀取 raw API tables 與治理 tables 後產生評分、分層與 snapshot；不得硬編股票名單 |
| `audit_core_universe.py` | **Core Universe Post-Build Verification** | 核心股結果驗收主權 | 只驗收已產生的 snapshot；不得重算核心股名單 |

### 6.0A Derived Schema 欄位繼承原則 (Column Inheritance Rule)

凡引用既有 raw API table 欄位語意者，欄位名稱大小寫與 SQL 型別寬度必須繼承 `DATASET_REGISTRY`，治理專用欄位才可另立 snake_case。完整規則沿用 v5.4.19 6.0A 條文，本版不修訂。

### 6.1 ~ 6.6 法理 / 分層 / CoreScore / 治理欄位 / 反偏誤 / 銜接邊界

完整法理沿用 v5.4.19 條文；本版補明「治理藍圖」與「現行實作版本」之邊界，避免將 v0.1 metadata bootstrap 誤認為完整核心股選拔模型。核心原則為：**先建立選拔制度，再產生名單；先選資料條件，不先選股票代號。**

- **6.1 法理**：第一性原則、20/60/20、MBNRIC、DB-driven。
- **6.2 分層**：research_universe / core_universe / convex_universe / quarantine_universe。
- **6.3 CoreScore**：25% DataQuality + 25% LiquidityMass + 20% FundamentalGravity + 15% ThemeResonance + 10% InstitutionalFlow + 5% VolatilityControl − RiskPenalty。
- **6.4 治理欄位**：含 price_coverage_252d、revenue_coverage_24m、financial_coverage_8q 等 lookback 指標。
- **6.5 反偏誤**：禁未來函數、禁倖存者偏誤、**禁同步引擎硬編名單**、必須可觀測、必須可重算。
- **6.6 銜接邊界**：治理層不存 feature/label/model/prediction values。

**核心股建立標準流程**：

1. `core_universe_schema.py`：先建立核心股治理 tables，只負責 governance schema，不碰 `data_schema.py` 的 raw API schema。
2. 核心候選資料灌溉：先以 `TaiwanStockInfo` 產出的 `research_universe` 作為候選池；若 research membership 尚未提交，先由全市場股票清單物化候選池，不得直接對全市場全表硬灌。第一階段只灌預設 4 核心表（價格 / PER / 法人 / 融資融券）與 730 日觀察窗；第二階段再依 coverage 缺口逐表補 `TaiwanStockMonthRevenue`、`TaiwanStockFinancialStatements` 等必要資料。
3. `core_universe_builder.py`：依資料計算 CoreScore、分層、snapshot、membership、scores 與 revision log。
4. `audit_core_universe.py`：驗收 Universe 是否合理、可回測、無偏誤、無跨層污染。
5. `sovereign_sync_engine.py --universe research / convex / core`：核心候選資料灌溉先消費 `research_universe`，不得以 `--all` 對全市場全表起跑；待 coverage 摘要指出缺口後，才逐表補刷並收斂至 `convex_universe` / `core_universe` 精灌。同步引擎不得自行決定核心股名單。

**CoreScore v0.2 六層評分流程**：

1. **硬性資格過濾**：
   - 必須存在於 `TaiwanStockInfo`。
   - 排除資料缺漏嚴重標的。
   - 排除極低流動性標的。
   - 排除長期停牌、異常交易、風險過高標的。

2. **資料品質分數 (DataQuality)**：
   - 價格資料完整度。
   - 成交量資料完整度。
   - 月營收資料完整度。
   - 財報資料完整度。
   - API / DB 欄位一致性。

3. **流動性質量分數 (LiquidityMass)**：
   - 平均日成交金額。
   - 平均成交量。
   - 成交連續性。
   - 波動承載能力。
   - 價格跳空與極端缺口。

4. **基本面重力分數 (FundamentalGravity)**：
   - 月營收成長。
   - 獲利能力。
   - 現金流或財報穩定度。
   - PER 合理性。
   - 股利與財務韌性。

5. **主題共振分數 (ThemeResonance)**：
   - AI / 半導體 / 算力供應鏈。
   - 機器人 / 自動化。
   - 生技 / 醫療。
   - 能源 / 儲能。
   - MBNRIC 第六波主題。

6. **風險懲罰 (RiskPenalty)**：
   - 高波動但無流動性支撐。
   - 基本面惡化。
   - 資料缺失。
   - 籌碼異常。
   - 長期弱勢或左尾風險。

**CoreScore v0.2 最小輸入資料契約（8 類）**：

`core_universe_builder.py v0.2` 的輸入資料以 `scripts/core/data_schema.py` 的 `DATASET_REGISTRY` 為唯一 schema 來源。表名、欄位名與唯一鍵不得在 builder 內另行定義；若 API 回傳欄位與 DB 欄位不一致，必須先在 ingestion / schema 對齊層修正，不得在選股模型內以臨時欄位補丁處理。

| 類別 | 主資料表 | 最小欄位契約 | 主要用途 | v0.2 裁決 |

本條為 §6.5 第 3 條「禁止同步引擎硬編名單」之具體實作契約。所有模組（含基礎設施 library 與序列模組）若需取得「目前生效之核心股名單」，**必須**使用以下唯一 SQL 契約，**不得**自行另立 SELECT 來源：

```sql
SELECT m."stock_id"
FROM "core_universe_membership" m
JOIN "core_universe_snapshot" s
  ON s."snapshot_id" = m."snapshot_id"
WHERE s."status" = 'committed'
  AND m."core_tier" IN ('core_universe', 'convex_universe')
  AND s."as_of_date" = (
      SELECT MAX("as_of_date")
      FROM "core_universe_snapshot"
      WHERE "status" = 'committed'
  )
ORDER BY m."stock_id";
```

**契約規格**：

1. **唯一來源**：已 committed 的 `core` / `convex` 同步範圍必須來自 `core_universe_membership` JOIN `core_universe_snapshot`；**禁止**查 v5.2 時代之 `stocks` 表、直接以 `TaiwanStockInfo`（市場宇宙）取代 committed membership，或使用任何硬編清單。`TaiwanStockInfo` 只可在 research membership 尚未提交時作為候選池物化來源，不得作為最終核心同步來源。
2. **生效條件**：必須 JOIN `core_universe_snapshot` 並過濾 `status = 'committed'`；未提交的 `draft` snapshot 不得作為來源。
3. **時點選擇**：必須選 `MAX(as_of_date)` 之 committed snapshot；若需歷史時點重算（Walk-forward），caller 應接受 `as_of_date` 參數並改為等號比對。
4. **層級範圍**：預設回傳 `core_universe` ∪ `convex_universe`；`research_universe` 與 `quarantine_universe` 不在預設同步範圍，但若為研究宇宙分階段灌溉，caller 可透過 `tiers=('research_universe',)` 明示納入 stage 1 補刷。
5. **排序**：必須 `ORDER BY stock_id` 以保證冪等與可重算。
6. **空集處理**：若 SELECT 回傳 0 筆，caller 必須以 `warning` 處理（不得 silent succeed），並通知 lifecycle context。

**封裝義務**：

- 本契約**應由 `db_utils.get_core_stocks_from_db()` 封裝**，供所有 caller 共用，避免每個模組各自寫 SQL（違反 DRY 與 SSOT）。
- `db_utils.py v2.45` 已完成封裝，實作查詢來源為 `core_universe_membership` JOIN `core_universe_snapshot`，並以 `status='committed'` 與 `MAX(as_of_date)` 保證生效版本唯一。
- 未來若需擴充（如歷史時點 `as_of_date` 參數、層級篩選參數），應在 `get_core_stocks_from_db()` 增加 keyword arguments，仍維持單一 SQL 來源。

**例外條款**：

- `audit_core_universe.py` 為驗收稽核工具，**允許**直接讀取 `core_universe_membership` 全部 tier 以驗證 builder 產物完整性；但不得回填到 sync 引擎。
- `core_universe_builder.py` 為 selection authority 本身，產生 membership，當然不適用本契約。
