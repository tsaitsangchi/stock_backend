# Quantum Finance 系統實體架構與目錄規範 (v5.2)

本文件定義了專案目錄結構與 **領域驅動設計 (DDD)** 邊界上下文的映射關係。全系統開發應嚴格遵循此目錄規範，以確保模組化與可擴展性。

---

## 1. 核心代碼空間 (The Codebase: `/scripts`)

所有邏輯腳本統一存放於 `scripts/` 目錄下，依職責劃分為六大模組：

### 1.1 `core/` (基礎設施層)
*   **職責**：提供全系統通用的基礎服務，不包含具體交易邏輯。
*   **關鍵檔案**：
    *   `db_utils.py`: 資料庫連線、批量寫入與結構自癒。
    *   `path_setup.py`: 22 維全路徑定錨與環境變數加載。
    *   `finmind_client.py`: API 供應鏈認證與配額管理。
    *   `model_metadata.py`: 模型註冊與元數據稽核。

### 1.2 `ingestion/` (數據攝取 Context)
*   **職責**：對接外部數據源，執行數據抓取與初步標準化入庫。
*   **規範**：此層級腳本嚴禁包含技術指標計算，僅負責數據「搬運」。

### 1.3 `features/` (特徵工程 Context)
*   **職責**：基於 **Polars** 高效運算引擎，將原始數據轉化為模型輸入特徵。
*   **產出**：產出結果存存放於 `data/feature_store/`。

### 1.4 `models/` (訓練與推理 Context)
*   **職責**：整合 AI 模型的生命週期管理。
    *   **訓練**：讀取特徵，訓練權重並存入 `models/weights/`。
    *   **推理**：讀取最新特徵與權重，生成每日預測信號。

### 1.5 `evaluation/` (評估與觀測 Context)
*   **職責**：系統的「反省中心」。
    *   **對齊**：比對預測值與真實股價。
    *   **監控**：計算模型漂移指標 (PSI) 與誤差 (MAE/RMSE)。

### 1.6 `maintenance/` (維運與自癒)
*   **職責**：全系統健康診斷 (`check_system_health.py`)、資料清理、索引重建等 DevOps 任務。

---

## 2. 數據與資產實體層 (Persistence & Artifacts)

### 2.1 `data/` (數據實體)
*   `raw/`: 存放原始抓取的數據備份 (建議使用 Parquet 格式)。
*   `feature_store/`: 存放極速特徵矩陣。

### 2.2 `models/` (資產實體)
*   `weights/`: 各版本的模型權重檔。
*   `scalers/`: 數據標準化參數對象。

---

## 3. 輸出與治理層 (Output & Governance)

*   **`predictions/`**: 存放每日生成的預測信號報告。
*   **`evaluations/`**: 存放預測準確率與模型漂移的圖表報告。
*   **`reports/`**: 存放系統架構、技術白皮書與開發文件。
*   **`logs/`**: 全系統的 `journalctl` 與自定義運行日誌。

---

## 4. 核心配置機制

*   **`.env` (Project Root)**:
    *   全系統 **唯一啟始點 (SSOT)**。
    *   定義 22 個物理路徑變數，驅動 `core.path_setup`。
    *   管理 API Token 與資料庫憑證。

---

## 5. 開發者操作規範

1.  **路徑引用**：嚴禁在代碼中使用絕對路徑，必須透過 `from core import get_xxx_dir` 獲取。
2.  **腳本位置**：新增功能前，必須先判斷該功能屬於哪一個「上下文」，並將腳本放入對應目錄。
3.  **日誌標準**：所有任務必須接入 `record_lifecycle` 以維持可觀測性。
