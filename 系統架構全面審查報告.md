```python
# 建議在 add_fundamental_features 中加入：
def add_staleness_features(df: pd.DataFrame) -> pd.DataFrame:
    """資料陳舊度特徵：ffill 不能消除的不確定性"""
    for col, lag in [("eps_ttm", "financial_statements"), ("revenue_yoy", "month_revenue")]:
        if col in df.columns:
            # 計算距上次真實更新的天數
            last_update = df[col].notna() & df[col].ne(df[col].shift(1))
            df[f"{col}_staleness_days"] = (~last_update).astype(int).groupby(
                (~last_update).cumsum()
            ).cumcount()
    return df
```

- `evaluate_fold` 加入 `avg_net_return`（含交易成本），對齊 `FRICTION_CONFIG`

**問題 ①：`TRAINING_STRATEGY.use_global_backbone` 未整合**

從 `config.py` 看到 `TRAINING_STRATEGY = {"use_global_backbone": True, "finetune_local": True}`，但 `train_evaluate.py` 的 `run_walk_forward` 函式仍是單標的訓練邏輯，並未依 `SECTOR_POOLS` 實作 backbone 預訓練。這個配置目前形同虛設。

**問題 ②：`calibration` 的時序有效性未驗證**

```python
# 使用 CalibratedClassifierCV 進行 Isotonic calibration 是好的，但：
# 問題：sklearn 的 CalibratedClassifierCV 預設使用 cross_val_predict，
# 在時間序列資料中這等同於用未來資料校準 → 破壞 Walk-Forward 的嚴格性

# 應確認使用 cv=TimeSeriesSplit 而非預設的隨機 KFold
```

**建議：**
```python

calibrated = CalibratedClassifierCV(
    base_estimator=meta_learner,
    cv=TimeSeriesSplit(n_splits=5),  # 強制時間順序
    method='isotonic'

```

**問題 ③：Walk-Forward fold 的 test_window 重疊問題未文件化**

`WF_CONFIG` 中 `test_window = 126` 而 `step_days = 21`，代表相鄰 fold 的 test 集有約 105 天重疊。這在程式碼的 docstring 中有提及「重疊不等於洩漏」，但這會導致 OOF 機率在重疊天數上被多次覆蓋，最終選取哪個版本的 OOF 值需要明確定義。

**建議：** 在 OOF 彙整時，對重疊日期採用最後一個 fold 的預測值（最新、訓練資料最多），並在 log 中記錄覆蓋情況。

**問題 ①：衰減率（decay）公式在 IC 為負值時語義反轉**

```python
# 目前計算：
decay = (np.abs(recent_ic) - np.abs(mean_ic)) / np.abs(mean_ic)
# 問題：若 mean_ic = -0.03（負向因子），|mean_ic| = 0.03
# 若 recent_ic = -0.01（衰減），|recent_ic| = 0.01
# decay = (0.01 - 0.03) / 0.03 = -0.67 → 顯示「嚴重衰減」✅ 結果正確
# 但若 mean_ic = -0.03，recent_ic = -0.06（增強）
# decay = (0.06 - 0.03) / 0.03 = +1.0 → 顯示「增強」✅ 也正確
# 實際上此公式對負向因子的語義處理是正確的，但需要加強文件說明
```

**問題 ②：未整合至 `train_evaluate.py` 的 Walk-Forward 循環中**

`feature_analysis.py` 是獨立工具，未被 `train_evaluate.py` 調用。理想情況下，每個 Walk-Forward fold 結束後都應執行因子穩定性分析，以便在下一個 fold 動態剔除衰減因子。

**建議：** 在 `run_walk_forward` 中加入因子穩定性的 fold-level 追蹤：

### 2.11 `singularity_layout_simulator.py`——2026 奇點佈局模擬器

**架構符合度：⭐⭐☆☆☆（2/5）**

這個模組體現了 README 的「2026 導航點」精神，但實作有多個根本性問題。

**問題 ①：交易邏輯過於原始**

```python
pull_thresh = -10.0  # ❌ 針對 2330 的物理感應門檻（硬編碼）
if position == 0 and gravity_pull < pull_thresh and info_force > 0:
    position = (capital * 0.2) / price  # 進場
elif position > 0 and gravity_pull > 0:  # ❌ 任何 gravity_pull > 0 就出場
    capital += position * price * (1 - 0.003)  # 只扣 0.3%，缺少完整摩擦成本
```

- `pull_thresh = -10.0` 是為 2330 手動校準的，但函式用於迴圈 4 個標的（2330, 2308, 2454, 3661），閾值不通用
- 出場條件 `gravity_pull > 0` 過於簡單，會因單日雜訊頻繁觸發
- 沒有停損機制

**問題 ②：回測無法重現，因特徵計算依賴即時資料庫**

模擬器直接呼叫 `build_daily_frame` + `build_features`，在不同時間點執行會產生不同的結果（因為資料庫持續更新）。一個可信的回測模擬器應凍結特定時間截面的資料。

**問題 ③：與 `portfolio_backtest.py` 功能重疊但方法不一致**

兩個模組都在做「持倉模擬」，但使用不同的邏輯和假設，可能導致對系統表現的解讀產生矛盾。

**建議：** 將 `singularity_layout_simulator.py` 重構為 `portfolio_backtest.py` 的一個策略插件，而非獨立的模擬器：

```python
# 在 portfolio_backtest.py 中加入策略模式
class SingularityStrategy(BaseStrategy):
    """2026 奇點佈局策略：重力井底部建倉 + 動能噴發出場"""
    def generate_signals(self, df_feat: pd.DataFrame) -> pd.Series:
        entry = (df_feat["gravity_pull"] < self.pull_thresh) & (df_feat["info_force_per_mass"] > 0)
        exit_ = df_feat["gravity_pull"] > self.exit_thresh
        return self._signals_to_positions(entry, exit_)
```

## 參、跨模組系統性問題

### 3.1 模組間依賴關係未文件化

目前存在的關鍵依賴鏈：

```
portfolio_strategy.py
    → signal_filter.py (SignalFilter)
    → feature_engineering.py (build_features)
    → data_pipeline.py (build_daily_frame) ← 未上傳，核心黑箱

portfolio_optimizer.py
    → data_pipeline.py (load_exchange_rate)
    → stock_forecast_daily 資料表 ← 由 predict.py 寫入

model_health_check.py
    → stock_forecast_daily 資料表
    → MODEL_DIR 的 .pkl 檔案
    ← 完全不知道 parallel_train.py 是否已訓練完成
```

**建議：** 建立 `ARCHITECTURE.md` 描述模組依賴 DAG，以及每個模組的輸入/輸出契約。

### 3.2 異常處理策略不一致

| 模組 | 錯誤處理策略 |
|---|---|
| `signal_filter._load_dynamics_registry` | `except Exception: pass`（靜默失敗） |
| `portfolio_strategy.main` | `except Exception as e: logger.error(...)` |
| `parallel_train.train_one_stock` | 返回 `(stock_id, False, str(e))` |
| `portfolio_optimizer.get_liquidity_multiplier` | 返回預設值 1.0 |

靜默失敗（`pass`）是最危險的模式，應改為至少記錄警告並返回明確的降級值。

### 3.3 測試覆蓋率為零

沒有任何 `tests/` 目錄或 `test_*.py` 檔案。對於一個生產量化系統，以下功能應有單元測試：

- `calculate_psi`：邊界值、全相同分佈（PSI=0）、完全不同分佈（PSI很大）
- `purged_walk_forward_folds`：確認 embargo 確實排除了未來資料
- `DATA_LAG_CONFIG` 應用：確認特徵計算後的日期對齊

### 3.4 `stock_dynamics_registry` 資料表未文件化

`signal_filter.py` 試圖從 `stock_dynamics_registry` 資料表讀取 `info_sensitivity`、`gravity_elasticity`、`fat_tail_index` 等欄位，但此資料表在 README 的 PostgreSQL 資料表對應清單中完全未提及。這是系統的隱性依賴，若資料表不存在，會靜默返回預設值並繼續運行，難以發現。
