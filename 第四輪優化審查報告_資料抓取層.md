## 貳、架構性問題：10 個腳本的嚴重程式碼重複

### 2.1 🔴 `finmind_get` 函式：10 種實作，10 種行為

每個 fetch 腳本都有自己獨立的 `finmind_get`，且重試邏輯互不相同：

| 腳本 | 重試機制 | 指數退避 | 連線逾時設定 | 400 處理 |
|---|---|---|---|---|
| `fetch_chip_data.py` | ✅ 有（3次） | ✅ `delay × 2^n` | ✅ (15, 120) | ✅ 400 直接跳過 |
| `fetch_technical_data.py` | ✅ 有（3次） | ❌ 固定 `delay×3` | ✅ (15, 120) | ❌ 未處理 |
| `fetch_derivative_sentiment_data.py` | ✅ 有（MAX次） | ❌ 固定延遲 | ✅ 分開處理 | ❌ 未處理 |
| `fetch_macro_data.py` | ❌ 無重試 | ❌ 無 | ❌ 60s 全局 | ❌ 未處理 |
| `fetch_total_return_index.py` | ❌ 無重試 | ❌ 無 | ❌ 30s 全局 | ❌ 未處理 |
| `fetch_international_data.py` | ✅ 有（3次） | ❌ 固定延遲 | ✅ (15, 120) | ❌ 未處理 |

**後果：** `fetch_macro_data.py` 碰到任何非 402 的網路問題（如 timeout）直接返回空資料，不重試，利率/匯率資料可能靜默缺失。

**根本解法：建立共享 API 層 `scripts/finmind_client.py`**

```python
# scripts/finmind_client.py — 唯一的 FinMind API 客戶端
import time, logging, requests
from config import FINMIND_TOKEN

FINMIND_API_URL = "https://api.finmindtrade.com/api/v4/data"
logger = logging.getLogger(__name__)

def wait_until_next_hour():
    """402 配額限制：等待至下一整點"""
    from datetime import datetime, timedelta
    now = datetime.now()
    next_hour = (now + timedelta(hours=1)).replace(minute=0, second=0, microsecond=0)
    wait_sec = (next_hour - now).total_seconds() + 65
    logger.warning(f"API 配額達上限（402），等待 {wait_sec:.0f} 秒至 {next_hour.strftime('%H:%M')} 恢復...")
    time.sleep(wait_sec)

def finmind_get(
    dataset: str,
    params: dict,
    delay: float = 1.2,
    max_retries: int = 3,
) -> list:
    """
    統一的 FinMind API 請求函式。
    重試策略：指數退避（delay × 2^n），最多 max_retries 次。
    402 處理：無限等待至下一整點後重試。
    400 處理：直接返回空列表（參數錯誤，無需重試）。
    """
    headers = {"Authorization": f"Bearer {FINMIND_TOKEN}"}
    req_params = {"dataset": dataset, **params}

    for attempt in range(1, max_retries + 1):
        try:
            resp = requests.get(
                FINMIND_API_URL, headers=headers,
                params=req_params, timeout=(15, 120)  # 連線15s, 讀取120s
            )
            # 402：配額耗盡，等待重置
            if resp.status_code == 402:
                wait_until_next_hour()
                continue  # 重試，不計入 attempt 次數

            # 400：請求參數錯誤，不重試
            if resp.status_code == 400:
                logger.debug(f"[{dataset}] 400 Bad Request，跳過（data_id={params.get('data_id')}）")
                return []

            resp.raise_for_status()
            payload = resp.json()

            if payload.get("status") == 402:
                wait_until_next_hour()
                continue

            if payload.get("status") != 200:
                logger.warning(f"[{dataset}] status={payload.get('status')}, msg={payload.get('msg')}")
                return []

            time.sleep(delay)
            return payload.get("data", [])

        except requests.exceptions.Timeout:
            logger.warning(f"[{dataset}] 第 {attempt}/{max_retries} 次逾時")
        except requests.exceptions.ConnectionError:
            logger.warning(f"[{dataset}] 第 {attempt}/{max_retries} 次連線失敗")
        except Exception as exc:
            logger.warning(f"[{dataset}] 第 {attempt}/{max_retries} 次異常：{exc}")

        if attempt < max_retries:
            backoff = delay * (2 ** attempt)  # 指數退避
            logger.info(f"[{dataset}] {backoff:.1f}s 後重試...")
            time.sleep(backoff)
        else:
            logger.error(f"[{dataset}] 已重試 {max_retries} 次，放棄。")
            return []
```

此後，每個 fetch 腳本只需：
```python
from finmind_client import finmind_get
```

一行取代 50~80 行重複程式碼，且所有腳本立即擁有一致的指數退避重試邏輯。

### 2.2 🔴 `fetch_total_return_index.py`：Token 傳遞方式錯誤


## 參、parallel_fetch.py 的三個關鍵問題

### 3.1 🟠 腳本路徑使用相對路徑，在非正確工作目錄時全部找不到

```python
# parallel_fetch.py
PHASE_1 = [
    "fetch_technical_data.py",   # ❌ 相對路徑
    "fetch_chip_data.py",
    ...
]

def run_script(script_path: str):
    if not os.path.exists(script_path):  # 在 scripts/ 以外的目錄執行時，全部是 False
        logger.error(f"找不到檔案: {script_path}")
        return script_path, False
```

若從專案根目錄執行 `python scripts/parallel_fetch.py`，`os.path.exists("fetch_technical_data.py")` 為 `False`，全部腳本都會被跳過且不執行。

**修復：**
```python
SCRIPTS_DIR = Path(__file__).parent  # 腳本所在目錄，無論從哪裡執行都正確
PHASE_1 = [
    str(SCRIPTS_DIR / "fetch_technical_data.py"),
    str(SCRIPTS_DIR / "fetch_chip_data.py"),
    ...
]
```

### 3.2 🟠 `subprocess.run` 沒有 timeout，單個卡死腳本會永久阻塞 Worker

```python
subprocess.run([VENV_PYTHON, script_path], env=env, check=True)  # ❌ 無 timeout
```

若 `fetch_fundamental_data.py` 在等待 402 重置時掛在 `time.sleep(wait_sec)` 長達一小時，這個 Worker 會被佔用一小時，阻塞後續腳本。

**修復：**
```python
MAX_SCRIPT_TIMEOUT = 3 * 3600  # 單腳本最長 3 小時
subprocess.run(
    [VENV_PYTHON, script_path], env=env,
    check=True, timeout=MAX_SCRIPT_TIMEOUT
)
```

### 3.3 🟡 三個 Worker 同時呼叫 FinMind API，配額耗盡速度是串行的 3 倍

FinMind 限制為 600 次/小時。當 3 個腳本並行時，實際上是 3 條 API 請求流同時消耗配額。若各腳本各需 250 次請求，並行執行約 23 分鐘就耗盡配額，而串行執行需要 75 分鐘才會耗盡。

觸發 402 後，3 個 Worker 會各自進入等待，且等待計時器不協調（可能 A 等到 :00，B 等到 :02，C 等到 :07）。

**建議：** 加入跨進程的 API 使用量監控，在開始前預查配額：

```python
# finmind_client.py 中加入
def check_api_quota() -> tuple[int, int]:
    """
    查詢當前 API 使用量與上限。
    回傳 (used, limit)。
    FinMind 文件：GET https://api.web.finmindtrade.com/v2/user_info
    """
    resp = requests.get(
        "https://api.web.finmindtrade.com/v2/user_info",
        headers={"Authorization": f"Bearer {FINMIND_TOKEN}"},
        timeout=10
    )
    data = resp.json()
    return data.get("user_count", 0), data.get("api_request_limit", 600)

# parallel_fetch.py main() 開始前
used, limit = check_api_quota()
remaining = limit - used
logger.info(f"API 配額：已用 {used}/{limit}，剩餘 {remaining} 次")
if remaining < 100:
    logger.warning("API 配額不足 100 次，建議等待下一個整點重置後再執行。")
    sys.exit(1)
```

### 4.2 🟡 `fetch_macro_data.py`：最弱的重試機制，負責最關鍵的宏觀資料

`fetch_macro_data.py` 抓取 FED/BOJ/ECB 利率、USD/TWD 匯率、美債殖利率，這些是 `feature_engineering.py` 的宏觀因子特徵的直接來源。但它的 `finmind_get` 是 10 個腳本中實作最脆弱的：

```python
def finmind_get(dataset: str, params: dict, delay: float = 1.2) -> list:
    while True:
        try:
            resp = requests.get(..., timeout=60)
            if resp.status_code == 402:
                wait_until_next_hour()
                continue
            resp.raise_for_status()
            ...
        except requests.exceptions.RequestException as e:
            # ❌ 任何非 402 的網路錯誤（timeout, connection reset）直接返回空列表，無重試
            if isinstance(e, requests.HTTPError) and e.response.status_code == 402:
                wait_until_next_hour()
                continue
            logger.error(f"[{dataset}] 請求失敗: {e}")
            return []  # ❌ 靜默失敗，利率資料可能為空
```

這意味著若 FinMind 伺服器有短暫抖動（10 秒連線失敗），FED 利率當日就不更新，但系統不會告警，模型繼續用過時利率計算 `fed_rate_chg_30d` 特徵。

### 4.3 🟡 `fetch_derivative_sentiment_data.py` 的裸 `except:` 語法

```python
def safe_float(v):
    try: return float(v)
    except: return None  # ❌ 裸 except，捕捉 KeyboardInterrupt/SystemExit

def safe_int(v):
    try: return int(v)
    except: return None  # ❌ 同上
```

裸 `except:` 會捕捉 `KeyboardInterrupt`，導致用 Ctrl+C 中斷腳本時無效，進程無法被正常終止。應改為 `except (TypeError, ValueError):`。

### 4.6 🟡 `fetch_technical_data.py` 的批次模式閾值設計問題

`fetch_dataset_batch` 的設計是「若同一起始日的股票數 ≥ batch_threshold → 一次抓全市場」，這是高效率的設計。但有個隱性問題：

```python
# 不帶 data_id 的全市場請求，回傳全市場所有標的的資料
# 然後只寫入 valid_stock_ids 中的標的
data = finmind_get("TaiwanStockPrice", {"start_date": s, "end_date": e})
# 過濾：只保留 valid_stock_ids 中的股票
rows = [map_price_row(r) for r in data if r.get("stock_id") in valid_set]
```

若台股有 1,700+ 標的，全市場請求回傳的資料量可能超過 FinMind 單次 API 的回傳上限（通常是 1,000 筆或按大小限制）。當資料被截斷時，`valid_stock_ids` 中靠後的標的可能靜默缺失，且無任何警告。

**建議：** 加入回傳筆數的閾值警告：
```python
BATCH_RETURN_WARNING_THRESHOLD = 5000
if len(data) >= BATCH_RETURN_WARNING_THRESHOLD:
    logger.warning(
        f"[{table}] 全市場批次請求回傳 {len(data)} 筆，"
        f"可能已達 API 回傳上限，部分資料可能被截斷！"
        f"建議降低 chunk_days 或使用逐支請求模式。"
    )
```
